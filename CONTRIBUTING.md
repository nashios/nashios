# Contributing

When contributing to this repository, please first discuss the change you wish to make via issue, email, or any other method with the owners of this repository before making a change.

On your first couple of contributions, start with something small. This will help you get familiar with the codebase and the contribution process.

## Building

You can find the instructions for building the project in the [build-instructions.md](documentations/build-instructions.md) file.

## Issues

Issues are a great way to get started. They are also a great way to get your ideas heard. If you have an idea for a new feature or a change to the existing codebase, please open an issue. This will allow us to discuss the idea and come up with a plan to implement it.

That being said, keep the following in mind:

- Search for existing issues. Someone may have already reported the issue you are facing.

- Be clear about what the issue is. Include as much information as possible. This will help us reproduce the issue and fix it faster.

- Include screenshots and animated GIFs. A picture speaks a thousand words.

- Open one issue per problem. Do not club multiple unrelated problems into a single issue.

- Jokes or irrelevant content are not welcome here. Please be respectful towards others.

## Language

We use English for all communications. Please use English when opening issues or submitting pull requests. This allows us to keep the repository consistent and understandable by everyone.

Feel free to use online translation services, spell checkers, and grammar checkers. They are your friends.

## Code style

Here is the list of good practices we follow:

- Follow the project coding style, you can find it in the [conding-style.md](documentations/coding-style.md) file.

- Use expressive names for variables, functions, structures, etc.

- Commit messages should be 72 characters limit and must follow the [commit-style.md](documentations/commit-style.md) format.

- Add your personal copyright line to the top of each file when making substantial changes.

And here is the list of bad practices we avoid:

- Don't submit code that is incopatible with the project [license](LICENSE).

- Change anything that is not related to your pull request.

- Don't add commented-out code.

- Write code in another language than C, this project is C-only and we want to keep it that way.
